# Route Catalog 100+100 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build and deploy a source-reviewed HikeSafe catalog containing the canonical 100 Taiwan Hundred Peaks and 100 reviewed suburban routes.

**Architecture:** Keep the existing JSON catalog, source registry, Zod validation, Drizzle importer, and PostgreSQL schema. Add a canonical 100-name suburban list as data, make the validator enforce that list and count, then populate the catalog in three reviewed data batches before importing it into Supabase.

**Tech Stack:** Node.js 24.x, TypeScript, Zod, Vitest, Next.js, Drizzle ORM, PostgreSQL, Vercel.

## Global Constraints

- Use Node 24.x and npm.
- Use official sources only; do not invent distance, elevation, duration, checkpoints, evacuation points, or permit information.
- Do not substitute an official "高度落差" value for cumulative ascent, calculate ascent from official KML, or assign a GIS rescue point by proximity unless the publishing authority explicitly defines that field or association.
- Keep `data/routes/catalog.json` and `data/routes/sources.json` valid JSON and preserve the existing `RouteInput` schema.
- Keep existing migration files immutable; this work does not require a database migration.
- Do not import into Supabase until `npm run routes:verify` reports a valid 100+100 catalog.
- Preserve route versioning, idempotent import behavior, and source/review metadata.

### Field-completeness gate discovered during execution

The Forestry official pages/API provide distance, entrance, duration class, difficulty class, permit status, altitude range, height difference, and KML. They do not publish cumulative ascent or an authority-defined evacuation-point list for each route. The National Parks GIS layer publishes rescue helipads and communication points, but does not associate them with catalog routes. Before Task 3 or 4 adds a record, either:

1. find an official source that explicitly supplies the missing field and route association; or
2. obtain approval to change the product data contract so unsupported fields are represented as unavailable rather than inferred.

The candidate source inventory and route-name list are not evidence that all 100 records meet this gate.
