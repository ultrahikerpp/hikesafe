# HikeSafe 快速建立行程設計

日期：2026-07-18

## 目標

讓使用者在保留已驗證路線、留守人與時間確認的前提下，快速建立新的單人行程。重複使用者應能以「使用上次路線」、確認本次時間與留守人後建立草稿；首次使用者仍使用同一個畫面的完整欄位。

本功能不提供自訂路線、導航、背景定位或自動救援。沒有已啟用 route version 時，系統仍不得建立行程。

## 使用流程

`/trips/new` 改為單頁快速建立表單。

1. 系統平行載入已啟用路線、目前有效的留守綁定，以及使用者最近一次已建立行程的快速預填資料。
2. 路線欄位預設未選取。若最近路線仍為已啟用版本，顯示「使用上次路線」操作；使用者必須主動點選，才會選定該路線。
3. 選定路線後，以該 route version 的官方 `durationMinutes` 和本次出發時間計算預計下山時間。出發時間預設為現在，精確到分鐘；使用者可覆寫預計下山時間，且後續操作不得覆寫其手動調整值。
4. 最近一次行程的有效留守綁定、車輛、裝備與領隊電話會預填。使用者可在同一頁修改；車輛、裝備與電話位於可展開的「行程與緊急資料」區。
5. 使用者需至少選擇一位目前有效的留守人，並勾選「我已確認路線、預計下山時間與留守人」後才能送出。沒有可用留守綁定時，現有綁定碼流程留在表單中。
6. 建立成功後維持既有行為，導向新草稿行程頁面。

完整設定不使用另一個建立 API；它只是同一頁表單中可展開的欄位。

## 資料與 API

新增受 LINE session 保護的快速預填讀取端點，例如 `GET /api/trips/quick-defaults`。

端點只讀取 session 使用者最近建立的一筆行程，並回傳：

- 最近使用的 route version ID，僅當該版本目前仍為 `isActive`。
- 最近行程中仍屬該使用者、已綁定且未撤銷的留守 binding IDs。
- 車輛、裝備與領隊電話。

端點不回傳失效路線、撤銷或未完成綁定，也不讀取其他使用者的行程。沒有歷史資料時回傳空預填資料，不是錯誤。

不新增偏好設定資料表或 browser localStorage。既有 `trips` 紀錄是唯一的「使用者已確認」資料來源；成功建立的新行程自然成為下一次預填來源，避免共用裝置殘留車輛與聯絡資料。

既有 `POST /api/trips` 與 `createTrip` 保持為唯一寫入流程，持續驗證已啟用 route version、綁定有效性、時間順序與 idempotency key。快速預填與前端確認不取代伺服器授權或驗證。

## 失敗行為

- 路線 catalog 無法讀取時，顯示既有安全訊息並禁止建立。
- 快速預填端點失敗時，表單仍可供首次使用者輸入；不顯示他人的資料，也不阻塞已驗證路線載入。
- 上次路線停用或上次留守綁定失效時，不預填相應選項，使用者必須重新選擇。
- 建立請求回傳綁定失效或路線停用時，顯示伺服器錯誤並重新載入可用綁定與路線；不把草稿宣稱為已建立。

## 範圍與限制

- 本次只調整單人行程建立；小隊邀請、副領隊與開始登山流程維持現況。
- 使用者仍需選擇或確認本次的已驗證路線、下山時間與留守人；系統不得靜默沿用上一趟資料。
- 緊急資料只會在使用者成功建立行程後成為下一次預填資料。

## 驗收與測試

- 快速預填端點拒絕未登入請求，且只能回傳自己的最近行程資料。
- 停用的 route version、撤銷或未綁定的留守人不會出現在預填回應中。
- 表單只有在路線已明確選定、至少一位有效留守人已選擇、時間有效且確認勾選後才可送出。
- 路線選擇會依官方預估時間填入結束時間；使用者手動修改後不會被覆寫。
- 快速預填不可用時，使用者仍能從空白表單建立行程；catalog 不可用時則仍不能建立。
- 保留既有行程建立 API、冪等與授權測試，並執行 `npm test` 和 `npm run build`。
